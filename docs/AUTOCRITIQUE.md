# Autocritique Adjugo — forces, faiblesses, risques

> Regard honnête multi-agents (15/06/2026), incluant la comparaison avec TenderCrunch.

## IA / RAG vs TenderCrunch

_Adjugo's IA layer lacks core TenderCrunch capabilities: NO RAG with source traceability, NO segmented multi-agent orchestration, NO questionnaire auto-completion with field mapping, NO post-generation compliance checking, and NO bi-modal output (narrative + RFP cell-by-cell). Current architecture is single-pass LLM analysis without persistent knowledge, inter-agent handoff, or structured field filling. Key gaps: (1) P1 - No vector DB or clickable source attribution in analysis (RAG missing); (2) P1 - Flat sequential pipeline, not segmented agents with specialization; (3) P1 - Cannot parse/auto-fill RFP/RFI/DDQ questionnaires; (4) P2 - No compliance validation post-redaction; (5) P2 - No dual output mode (long-form + Q&A); (6) P2 - Limited hallucination audit trail; (7) P2 - Hardcoded French prompts, no country-specific specialization; (8) P2 - No streaming agentic status feedback. Below: 9 prioritized findings with exact file paths and implementation guidance._

- **[P1] RAG Architecture & Knowledge Base Traceability** (app/services/llm.py:1-69, app/services/analysis.py:237-301, app/routers/sourcing.py:353-388)
  - Problème : NO vector database or embedded knowledge base. The LLM analysis layer (app/services/analysis.py, app/services/llm.py) relies entirely on prompt-based analysis of raw DCE text, with zero persistent context management or source attribution. Each analysis call extracts and reprocesses raw document text without semantic indexing. Zero 'clickable source' links back to DCE clauses, unlike TenderCrunch's RAG with absolute source traceability.
  - Reco : Implement a production-grade RAG pipeline: (1) Vectorize every clause/paragraph of uploaded DCE + BOAMP/TED notices with embeddings (Anthropic's text-embedding-3 or Claude's retrieval API); (2) Store embeddings in a vector DB (Pinecone/Supabase pgvector) with metadata (doc_id, page, clause_type, acheteur); (3) For each analysis question or field fill, retrieve top-K relevant clauses with exact byte offsets; (4) Inject retrieved clauses as context into LLM with source attribution; (5) In response JSON, include 'source_excerpts': [{clause_text, document_path, offset_start:offset_end}] for UI link-back to original PDF. File paths: app/services/llm.py:init RAG client, app/services/analysis.py:refactor analyze_dce_text() to call retrieval, app/services/rag_store.py (NEW), app/routers/sourcing.py:enhance /ask endpoint with source links.
- **[P1] Multi-Agent Pipeline Segmentation** (app/services/agents/__init__.py:1-2, app/routers/sourcing.py:1-450, app/services/agents/redaction.py:1-70)
  - Problème : Four agents named in memory (sourceur, Go/No-Go strategiste, groupement, redacteur) but actual code shows NO segmented multi-agent orchestration. Current flow is flat sequential: (1) search_tenders() → (2) analyze_tender() calls ONE LLM instance analyzing DCE broadly → (3) source_cotraitants() (deterministic API, no LLM) → (4) generate_documents() calls ONE LLM for memoire. Each step is independent; no agent-to-agent message passing, no orchestrator, no specialization per agent. TenderCrunch's architecture: extraction specialist → plan specialist → redaction specialist → compliance checker, each with fine-tuned system prompts and memory handoff.
  - Reco : Architect a proper multi-agent orchestration system: (1) Agent 1 'Sourcing': score & rank raw BOAMP/TED notices (already exists in TenderSearchService); (2) Agent 2 'Go/No-Go Analyst': parse DCE structure, extract criteria, benchmark vs company profile (currently in analyze_dce_text but monolithic); (3) Agent 3 'Groupement Optimizer': decompose lots, recommend co-traitant composition by trade/dept (partially in /groupement endpoint, needs refinement); (4) Agent 4 'Redacteur': generate CERFA + memoire technique + questionnaire auto-fill (currently in redaction.py). Implement: (1) app/services/agents/orchestrator.py (NEW) with async task queue (Celery/RQ) or in-process state machine; (2) Define agent message types (AgentInput/AgentOutput dataclasses); (3) Enhance each agent with explicit system prompts in app/services/agents/{analyst,optimizer,redactor}.py; (4) Add inter-agent context passing via Project.workflow JSON; (5) Expose orchestration status via /api/sourcing/{project_id}/status (streaming).
- **[P1] Questionnaire Auto-Completion & RFP/RFI/DDQ Field Mapping** (app/services/cerfa.py:90-205, app/routers/sourcing.py:420-450, app/services/agents/redaction.py:27-59 (needs refactor))
  - Problème : NO questionnaire auto-completion or structured field filling. The system generates static CERFA PDFs (DC1, DC2, ATTRI1) with pre-filled company data (app/services/cerfa.py:90-205), but: (1) Zero parsing of customer RFP/RFI/DDQ spreadsheets or structured question lists; (2) Zero mapping of DCE content to RFP/RFI questions (e.g., 'capacity' section → RFP question 12); (3) Zero cell-by-cell auto-completion logic; (4) No tracking of which answers come from DCE vs company profile vs defaults. TenderCrunch's differentiator: bi-modal output (memory long + cellule-par-cellule RFP answering with source attribution).
  - Reco : Build a questionnaire engine: (1) Create app/services/questionnaire.py (NEW) with SchemaParser class to ingest RFP/RFI/DDQ files (Excel, PDF forms, JSON); (2) Normalize questions to a QuestionType enum (capacity, experience, methodology, compliance, price, delivery, personnel, insurance, etc.); (3) Create AnswerMatcher class that maps DCE clauses/extracted data (via RAG) → RFP questions with confidence scores; (4) Implement FieldFiller agent that generates natural-language answers contextualized to each question + source citations; (5) Add endpoint POST /api/sourcing/{project_id}/autofill-questionnaire that takes uploaded RFP + returns {'question_id': {'answer': str, 'confidence': float, 'sources': [...]}}; (6) Expose /api/sourcing/{project_id}/questionnaire for UI to display fillable grid with AI-suggested answers + confidence indicators. Example: RFP row 'Describe your team structure' → FieldFiller extracts from DCE 'moyens humains' section + company qualifications → returns answer 'Équipe de 12 ingénieurs spécialisés...' [source: mémoire technique p.2].
- **[P2] Compliance Checker Agent (4th Agent)** (app/routers/sourcing.py:420-450, app/services/agents/redaction.py:61-143)
  - Problème : No post-redaction compliance validation. The Redacteur generates memoire + CERFA but no agent verifies: (1) All mandatory clauses answered (from DCE 'pieces_requises'); (2) Consistency between CERFA sections (D,E,F) and company profile; (3) Risk clauses acknowledged (penalites, garantie, delai) vs company thresholds; (4) Response completeness score (e.g., 87% fields filled, 3 critical gaps flagged). TenderCrunch: conformance checker catches missing DUME sections, unsigned attestations, mismatches.
  - Reco : Create app/services/agents/compliance.py (NEW) with a ComplianceChecker class: (1) Parse DCE 'pieces_requises' list; (2) Cross-check generated documents against list (DC1, DC2, ATTRI1, memoire, national form if EU); (3) Validate CERFA section consistency (e.g., company name in D matches in company profile, all co-traitants listed in E are in groupement); (4) Verify risk clauses (loop through ai_analysis['details']['clauses_risque'], confirm user acknowledged each 'eleve' level); (5) Calculate completeness score (fields_filled/fields_total); (6) Return ComplianceReport with {missing_docs: [], consistency_errors: [], risk_gaps: [], completeness_pct: int, must_fix: [], nice_to_have: []}. Add endpoint POST /api/sourcing/{project_id}/validate-compliance returning report + auto-generated remediation checklist. Call this before /documents returns download link.
- **[P2] Long-Form Memory vs Questionnaire Bi-Modal Output** (app/services/agents/redaction.py:27-59, app/routers/sourcing.py:427-445)
  - Problème : Redacteur generates single-format output: memoire Markdown (app/services/agents/redaction.py:27-58) + CERFA PDFs. No dichotomy between: (1) 'long-form strategic memory' (narrative, continuous prose answering 'how will you execute?'); (2) 'questionnaire-driven cell-by-cell responses' (structured, indexed, with source traceability). A single RFP might need both: marketing material (memoire) + technical detail sheets (RFP Q&A grid).
  - Reco : Refactor redaction agent to produce dual output: (1) Maintain existing memoire_technique as 'strategic narrative'; (2) Add new generation mode POST /api/sourcing/{project_id}/generate-questionnaire-answers: takes optional RFP template, outputs {questions: [{q_id, q_text, a_text: str, a_sources: [clause_ref], a_confidence: float, a_length_chars: int}]}. Use different temperature + length constraints for each mode (memoire: temp=0.4, ~600 words; QA: temp=0.1, ~150 words per Q). Inject RAG context differently: memoire gets full DCE context; QA gets only relevant clauses per question. File: app/services/agents/redaction.py refactor _generate_memoire_fast() into _generate_memoire_strategic() + new _generate_questionnaire_answers() + shared helper _retrieve_context().
- **[P2] Hallucination Prevention & Document Integrity Audits** (app/services/analysis.py:237-301, app/services/llm.py:29-68)
  - Problème : Anti-hallucination claims in code comments (e.g., app/services/agents/amont.py:8-11 'jamais invente', 'traçabilité') but no runtime verification or audit trail. No systematic checks: (1) Does extracted project budget match actual DCE text? (2) Are clauses_risque values cited in the document or fabricated? (3) Do generated CERFA field values come from uploaded company data or invented? No audit log of LLM inputs/outputs per call.
  - Reco : Implement strict source attestation: (1) app/core/llm_audit.py (NEW) wraps every complete/complete_json call to log {timestamp, model, input_tokens, output_tokens, input_hash, output_hash, user_id, project_id}; (2) Add SourceCitation model to track every claimed fact: {field_name, claimed_value, source_type: 'company_profile'|'dce_text'|'inferred'|'user_input', source_excerpt, confidence: float}; (3) In analyze_dce_text(), validate every extracted value against source text with regex/fuzzy matching before returning; (4) If value not found in DCE, mark as 'inferred' + lower confidence; (5) Export audit trail per project: GET /api/admin/audit/{project_id} → audit log + source attestations. Display 'confidence' badge in UI (green ✓ = found in doc, orange ⚠ = inferred, red ✗ = uncertain).

File: app/services/analysis.py refactor to call prompt_factory.get_system_prompt(country).

- **[P2] Qualité du Go/No-Go = arbitraire et non validé** (/Users/eliot/Downloads/adjugo-backend/app/services/analysis.py:13-87, app/routers/analysis.py:72-75, app/models/__init__.py:177-178)
