# Revue de code Adjugo — audit automatisé

> Audit multi-agents (15/06/2026). Priorités P1 (critique) → P3 (mineur). Plusieurs P1 corrigés dans la foulée (voir git log).

## Architecture & qualité

_The Adjugo backend is a well-structured FastAPI application for public procurement response automation. Code quality is generally good with clear separation of concerns, but several critical issues were identified in configuration security, error handling, database patterns, API endpoint hardcoding, and dependencies. The architecture follows FastAPI conventions effectively with proper use of routers, services, and models, but has weaknesses in production-readiness and robustness._

- **[P1] Configuration Security** (app/core/config.py:15)
  - Problème : Default SECRET_KEY hardcoded as 'change-this-in-production'. This insecure default will be used if not overridden in .env, making JWT tokens predictable and all authentication bypassable in production.
  - Reco : Remove the hardcoded default SECRET_KEY entirely (set to empty string) and raise a clear error at startup if ANTHROPIC_API_KEY and SECRET_KEY are not configured. Alternatively, generate a random key at first run and persist it.
- **[P1] LLM Client Initialization** (app/services/analysis.py:11)
  - Problème : Client instantiated at module import time: `client = Anthropic(api_key=settings.ANTHROPIC_API_KEY)`. If ANTHROPIC_API_KEY is empty (dev without API key), this fails silently or creates an invalid client. The llm.py service uses lazy initialization (better pattern) but analysis.py does not.
  - Reco : Use lazy initialization like in llm.py: create client only when needed, or validate ANTHROPIC_API_KEY at startup and fail fast with a clear error message.
- **[P1] Stripe Hardcoded URLs** (app/routers/stripe_pay.py:38-39, 101)
  - Problème : Stripe checkout success/cancel URLs hardcoded to 'http://localhost:5173'. This will break all Stripe payment flows in production, redirecting users to the wrong domain.
  - Reco : Use environment variables for frontend URLs (FRONTEND_URL / APP_BASE_URL). Construct URLs dynamically: `f'{settings.FRONTEND_URL}/?payment=success'`.
- **[P1] N+1 Query Pattern** (app/routers/cotraitants.py:129-130)
  - Problème : Loop-over-fetched-IDs pattern: queries all ProjectCotraitant links, then fetches each Cotraitant with individual .first() query inside the loop. With 100 project co-traitants, this executes 100+ queries instead of 1 JOIN.
  - Reco : Use eager loading or JOIN: `db.query(ProjectCotraitant).join(Cotraitant).filter(...).all()` and build dict in Python, or use sqlalchemy.orm.joinedload.
- **[P1] Silenced Error Handling** (app/routers/org.py:108-116)
  - Problème : Email send failures caught but silently swallowed with bare `except: pass`. If SMTP fails, users are not notified of invitation issues. No logging, no indication to caller.
  - Reco : Log all exceptions: `except Exception as e: logger.warning('Failed to send invite email: %s', e)`. If critical (non-best-effort), re-raise or return error indication.
- **[P1] Default Configuration Exposed** (app/core/config.py:24, 47)
  - Problème : Default values include localhost CORS origins and PostgreSQL credentials. If .env is missing, the app will be accessible from localhost:8000/5173 and connect to default DB, risking data exposure in misconfigured deployments.
  - Reco : Remove all default values except non-sensitive ones (ALGORITHM, LOG_LEVEL). Raise ValueError at startup if critical env vars are missing (SECRET_KEY, ANTHROPIC_API_KEY in prod).
- **[P2] Inconsistent Exception Handling** (app/core/quota.py:70-71)
  - Problème : In consume_analysis(), overage reporting failure is silently swallowed with bare `except: pass`. Billing errors are hidden from logs and developers.
  - Reco : Log all failures: `except Exception as e: logger.warning('Overage reporting failed for user %s: %s', user.id, e)`.
- **[P2] Duplicate LLM Client Initialization** (app/services/analysis.py:11 vs app/services/llm.py:21-26)
  - Problème : Two separate Anthropic client implementations: analysis.py creates client directly, llm.py uses lazy singleton pattern. Duplication means inconsistent retry logic, timeout handling, and makes it hard to manage API keys centrally.
  - Reco : Consolidate: have analysis.py use the llm.py.client() function. Remove direct Anthropic import from analysis.py. Unify all LLM calls through llm.py layer.
- **[P2] Hard-Coded httpx Version Constraint** (requirements.txt:13)
  - Problème : httpx pinned to 0.27.2 with comment explaining anthropic 0.34 incompatibility with newer httpx. This coupling is fragile: updating anthropic will require manual httpx downgrade. Future anthropic versions may fix this, leaving the constraint stale.
  - Reco : Monitor anthropic releases and test with latest httpx. Consider using a version range `httpx>=0.27,<0.30` or document a migration plan. Add pre-commit check to verify compatibility at CI.
- **[P2] No Soft-Delete Consistency** (app/routers/contacts.py, app/routers/cotraitants.py (hard delete) vs app/routers/documents.py (soft delete))
  - Problème : Mixed deletion strategies: Documents use soft-delete (deleted_at), but Contacts and Cotraitants use hard delete (db.delete()). Inconsistency makes recovery impossible for contacts/cotraitants, audit trails incomplete.
  - Reco : Standardize on soft-delete across all entities. Add deleted_at column to Contact and Cotraitant models. Update routes to filter deleted_at.is_(None).
- **[P2] Unclear Error Messages in LLM Parsing** (app/services/llm.py:68)
  - Problème : parse_json() raises generic ValueError with raw response text (up to 200 chars). When LLM returns unexpectedly, users see unhelpful raw JSON parse errors instead of a clear 'Analysis failed, try again'.
  - Reco : Return a structured error dict or raise a custom exception with context (model used, retry hint). Log the full response internally.
- **[P2] Database Connection Pool Configuration Missing** (app/core/database.py:20-26)
  - Problème : PostgreSQL engine configured with pool_pre_ping=True but no explicit pool_recycle. In production with long-lived connections, stale connections may not recycle, causing 'connection reset by peer' errors after ~15 mins idle.
  - Reco : Add pool_recycle=3600 (1 hour) to the PostgreSQL engine creation. This auto-recycles connections older than 1 hour.
- **[P2] Missing Database Migration on Startup** (app/main.py:25-31)
  - Problème : SQLite auto-creates schema via Base.metadata.create_all(), but PostgreSQL requires manual alembic migrations. Comment says migrations are required but no validation at startup checks if they've been applied. If deployed without running alembic, the app will crash on first query.
  - Reco : Add a startup check: run `alembic current` to verify migrations are up-to-date. Fail fast with clear error if not.
- **[P2] Missing Status Code Validation** (app/routers/stripe_pay.py:38-42)
  - Problème : Stripe session.url is returned directly without validating session creation actually succeeded. If Stripe fails silently, a None or invalid URL is returned to frontend.
  - Reco : Validate: `if not session.url: raise HTTPException(500, 'Payment session creation failed')`. This is especially important for financial transactions.
- **[P2] Async Scheduler Lacks Error Recovery** (app/main.py:136-159)
  - Problème : The amont_scheduler background task catches all exceptions and logs warnings but continues the loop. If a systematic error occurs (e.g., database connection lost), the loop will silently fail every iteration without alerting operators.
  - Reco : Add exponential backoff or circuit breaker: log errors once, then skip N iterations before retrying. Alert on repeated failures.
- **[P3] Test Coverage Gaps** (tests/ (only 4 files))
  - Problème : Only smoke tests and sourcing tests present. No coverage for: cerfa generation, agent services, complex quota logic, error paths, security (CSRF, XSS), concurrency.
  - Reco : Add unit tests for key services: analysis.py, cerfa.py, quota.py, storage.py. Add integration tests for auth flows, soft-delete consistency. Aim for >60% coverage of critical paths.
- **[P3] No Input Validation on File Uploads** (app/routers/documents.py:26-37, app/routers/analysis.py:32-36)
  - Problème : Files are validated by size and extension, but no content-type verification. A malicious .pdf file containing executable code could be stored and re-downloaded, bypassing browser sandbox.
  - Reco : Add MIME type verification: check magic bytes of uploaded file. For PDFs, verify %PDF- header. Reject files where declared MIME doesn't match detected type.
- **[P3] Unused Pass Statements** (app/routers/org.py:142 (likely), app/routers/documents.py, app/services/profile.py)
  - Problème : Bare `pass` statements in exception handlers indicate incomplete error handling or dead code paths. Makes intent unclear.
  - Reco : Replace with explicit logging or re-raise. If truly intentional, add a comment explaining why the error is suppressed.
- **[P3] Missing Rate Limit Configuration Validation** (app/core/ratelimit.py (inferred), app/core/config.py:27)
  - Problème : RATELIMIT_STORAGE_URI defaults to 'memory://'. In multi-worker deployments (gunicorn, uvicorn workers), each worker has its own in-memory store, so rate limits are not actually enforced globally.
  - Reco : In production, validate RATELIMIT_STORAGE_URI points to Redis. Add startup check: if workers > 1 and storage is memory:// warn or fail.
- **[P3] Timezone Handling Inconsistency** (app/models/__init__.py:55, app/core/config.py (missing TIMEZONE setting))
  - Problème : Timestamps use UTC (good), but no explicit TIMEZONE env var. Logs and reports may be confusing if users expect local time. No conversion for display.
  - Reco : Add TIMEZONE env var. Use pytz or zoneinfo to convert display times. Ensure all DateTime fields are timezone-aware.

## Sécurité

_Adjugo backend security audit: 13 findings across authentication, file handling, HTTP headers, rate limiting, admin endpoints, and cryptography. Three P1-severity issues require urgent remediation: hardcoded SECRET_KEY default, incomplete filename sanitization in exports (file_path reference bug + path traversal), and HTTP header injection via unvalidated Content-Disposition values. Six P2 issues include X-Forwarded-For spoofing, exposed API docs, weak cron authentication (especially in DEMO_MODE), and Stripe webhook signature bypass. Four P3 findings are lower-risk or already mitigated by frameworks but recommend incremental improvements (CSP hardening, password hashing upgrade, concurrent quota race condition)._

- **[P1] Authentication & Secrets Management** (app/core/config.py:15)
  - Problème : Hardcoded default SECRET_KEY in production-like config. Default value 'change-this-in-production' will be used if .env is not properly set.
  - Reco : Enforce SECRET_KEY validation at startup: raise error if SECRET_KEY contains 'change-this' OR is fewer than 32 characters. Add pre-flight checks in main.py startup hooks.
- **[P1] File Upload & Path Traversal** (app/routers/export_dossier.py:101-109)
  - Problème : export_dossier.py references non-existent 'doc.file_path' attribute (models use 'file_key'). This causes FileNotFoundError, but code gracefully skips documents. More critically, filename sanitization is incomplete - only replaces '/' and ' ' with '_', allowing potential directory traversal with '..\' on Windows or encoded variants.
  - Reco : 1) Fix attribute reference: use storage.load(doc.file_key) instead of os.path.exists(doc.file_path). 2) Use pathlib.Path.name or werkzeug.utils.secure_filename to sanitize filenames. 3) Validate filename length and reject suspicious patterns.
- **[P1] HTTP Response Header Injection (Content-Disposition)** (app/routers/documents.py:248, app/routers/cerfa.py:70,74, app/routers/export_dossier.py:168,173)
  - Problème : Filename values inserted directly into Content-Disposition headers without sanitization. User-controlled data (doc.name, project.name) can inject CRLF characters or quotes to break header parsing or trigger browser exploits.
  - Reco : Sanitize all filenames before header insertion: remove/escape quotes, CR/LF, and control characters. Use RFC 5987 encoding (filename*=UTF-8''...) for non-ASCII names.
- **[P2] Rate Limiting Spoofing (X-Forwarded-For)** (app/core/ratelimit.py:13-18)
  - Problème : X-Forwarded-For header trusted without validation. Attackers behind reverse proxy can spoof IPs by rotating XFF values to bypass rate limits.
  - Reco : If behind trusted proxy, configure proxy to strip incoming XFF and set server-side. Otherwise, whitelist proxy IPs or use x-real-ip. Document proxy trust assumptions in deployment guide.
- **[P2] Admin Endpoints - Weak Authentication** (app/routers/admin.py:18-25)
  - Problème : CRON_SECRET compared with '==' (non constant-time). More critically: in DEMO_MODE (currently TRUE by default), admin endpoints (/api/admin/run-alerts, etc.) require NO authentication, allowing public access to trigger alerts and AI detection in production.
  - Reco : 1) Use secrets.compare_digest for constant-time comparison. 2) Change DEMO_MODE default to FALSE with loud warning if True in production. 3) Require CRON_SECRET even in DEMO_MODE.
- **[P2] API Documentation Exposed** (app/main.py:33)
  - Problème : Swagger/OpenAPI docs exposed at /docs without authentication. Unauthenticated users can discover all endpoints, schemas, field names, and implementation details.
  - Reco : Disable docs in production (docs_url=None). If needed, protect with authentication or set docs_url=None when ENVIRONMENT='production'.
- **[P2] Hard-coded Stripe Redirect URLs** (app/routers/stripe_pay.py:38-39)
  - Problème : Stripe checkout success/cancel URLs point to localhost:5173 (Vue dev server), causing production redirects to fail if not updated.
  - Reco : Load success_url and cancel_url from settings (FRONTEND_URL). Validate at startup: raise error if FRONTEND_URL contains 'localhost' in production.
- **[P2] JWT Token Expiry Too Long** (app/core/config.py:17, app/core/security.py:33-44)
  - Problème : ACCESS_TOKEN_EXPIRE_MINUTES defaults to 1440 (24 hours), long for sensitive SaaS. No token revocation (blacklist/denylist) implemented, allowing old tokens until natural expiry.
  - Reco : Reduce default to 60 minutes (configurable). Implement token revocation in Redis. Return refresh tokens separately (short-lived access + longer-lived refresh).
- **[P2] Stripe Webhook - Weak Signature Verification** (app/routers/stripe_pay.py:116-120)
  - Problème : Webhook handler accepts unsigned events when STRIPE_WEBHOOK_SECRET is empty. Allows forged webhook events to modify subscription plans without validation.
  - Reco : Require STRIPE_WEBHOOK_SECRET at startup. Never fall back to unsigned JSON parsing - reject webhook if signature check fails. Log all webhook attempts.
- **[P2] Sensitive Data in API Responses** (app/routers/org.py:96-119)
  - Problème : Temp passwords returned in API responses during member invitation, exposing them in server logs, API response logs, and browser history.
  - Reco : Send temp password ONLY via email, never return in API response. Require user to set own password after first login (force reset). Add audit log.
- **[P2] CORS Configuration Too Permissive** (app/main.py:42-43, app/core/config.py:24)
  - Problème : CORS_ORIGINS allows multiple origins and falls back to hardcoded dev origins if env var missing. Compromised domain can make authenticated requests on behalf of users.
  - Reco : Require explicit CORS_ORIGINS in production. Validate each origin against whitelist. Avoid wildcard origins with allow_credentials=True.
- **[P3] Cron Secret Not Enforced in Prod** (app/core/config.py:44)
  - Problème : CRON_SECRET defaults to empty string. No enforcement prevents empty value in production, making cron endpoints accessible without secret.
  - Reco : Add startup validation: if ENVIRONMENT='production' and CRON_SECRET empty, raise error immediately. Document mandatory cron secret for prod.
- **[P3] CSP 'unsafe-inline' and 'unsafe-eval' Enabled** (app/main.py:51-59)
  - Problème : CSP allows 'unsafe-inline' and 'unsafe-eval' due to Vue.js CDN. Weakens XSS protection; documented as temporary pending build step migration.
  - Reco : Implement Vue 3 build step (Vite) to pre-compile templates and inline styles. Generate bundles with CSP hashes. Remove unsafe directives. Alternatively use nonce-based CSP short-term.
- **[P3] Password Storage - Bcrypt Recommended** (app/core/security.py:18-30)
  - Problème : PBKDF2 with 100k iterations meets OWASP but slower than bcrypt/argon2. No KDF context used, though salt regeneration prevents rainbow tables.
  - Reco : Migrate to bcrypt (bcrypt.hashpw) or argon2-cffi with higher cost factors. Update migration to hash old passwords on first login after upgrade.

## IA / RAG vs TenderCrunch

_Adjugo's IA layer lacks core TenderCrunch capabilities: NO RAG with source traceability, NO segmented multi-agent orchestration, NO questionnaire auto-completion with field mapping, NO post-generation compliance checking, and NO bi-modal output (narrative + RFP cell-by-cell). Current architecture is single-pass LLM analysis without persistent knowledge, inter-agent handoff, or structured field filling. Key gaps: (1) P1 - No vector DB or clickable source attribution in analysis (RAG missing); (2) P1 - Flat sequential pipeline, not segmented agents with specialization; (3) P1 - Cannot parse/auto-fill RFP/RFI/DDQ questionnaires; (4) P2 - No compliance validation post-redaction; (5) P2 - No dual output mode (long-form + Q&A); (6) P2 - Limited hallucination audit trail; (7) P2 - Hardcoded French prompts, no country-specific specialization; (8) P2 - No streaming agentic status feedback. Below: 9 prioritized findings with exact file paths and implementation guidance._

- **[P1] RAG Architecture & Knowledge Base Traceability** (app/services/llm.py:1-69, app/services/analysis.py:237-301, app/routers/sourcing.py:353-388)
  - Problème : NO vector database or embedded knowledge base. The LLM analysis layer (app/services/analysis.py, app/services/llm.py) relies entirely on prompt-based analysis of raw DCE text, with zero persistent context management or source attribution. Each analysis call extracts and reprocesses raw document text without semantic indexing. Zero 'clickable source' links back to DCE clauses, unlike TenderCrunch's RAG with absolute source traceability.
  - Reco : Implement a production-grade RAG pipeline: (1) Vectorize every clause/paragraph of uploaded DCE + BOAMP/TED notices with embeddings (Anthropic's text-embedding-3 or Claude's retrieval API); (2) Store embeddings in a vector DB (Pinecone/Supabase pgvector) with metadata (doc_id, page, clause_type, acheteur); (3) For each analysis question or field fill, retrieve top-K relevant clauses with exact byte offsets; (4) Inject retrieved clauses as context into LLM with source attribution; (5) In response JSON, include 'source_excerpts': [{clause_text, document_path, offset_start:offset_end}] for UI link-back to original PDF. File paths: app/services/llm.py:init RAG client, app/services/analysis.py:refactor analyze_dce_text() to call retrieval, app/services/rag_store.py (NEW), app/routers/sourcing.py:enhance /ask endpoint with source links.
- **[P1] Multi-Agent Pipeline Segmentation** (app/services/agents/__init__.py:1-2, app/routers/sourcing.py:1-450, app/services/agents/redaction.py:1-70)
  - Problème : Four agents named in memory (sourceur, Go/No-Go strategiste, groupement, redacteur) but actual code shows NO segmented multi-agent orchestration. Current flow is flat sequential: (1) search_tenders() → (2) analyze_tender() calls ONE LLM instance analyzing DCE broadly → (3) source_cotraitants() (deterministic API, no LLM) → (4) generate_documents() calls ONE LLM for memoire. Each step is independent; no agent-to-agent message passing, no orchestrator, no specialization per agent. TenderCrunch's architecture: extraction specialist → plan specialist → redaction specialist → compliance checker, each with fine-tuned system prompts and memory handoff.
  - Reco : Architect a proper multi-agent orchestration system: (1) Agent 1 'Sourcing': score & rank raw BOAMP/TED notices (already exists in TenderSearchService); (2) Agent 2 'Go/No-Go Analyst': parse DCE structure, extract criteria, benchmark vs company profile (currently in analyze_dce_text but monolithic); (3) Agent 3 'Groupement Optimizer': decompose lots, recommend co-traitant composition by trade/dept (partially in /groupement endpoint, needs refinement); (4) Agent 4 'Redacteur': generate CERFA + memoire technique + questionnaire auto-fill (currently in redaction.py). Implement: (1) app/services/agents/orchestrator.py (NEW) with async task queue (Celery/RQ) or in-process state machine; (2) Define agent message types (AgentInput/AgentOutput dataclasses); (3) Enhance each agent with explicit system prompts in app/services/agents/{analyst,optimizer,redactor}.py; (4) Add inter-agent context passing via Project.workflow JSON; (5) Expose orchestration status via /api/sourcing/{project_id}/status (streaming).
- **[P1] Questionnaire Auto-Completion & RFP/RFI/DDQ Field Mapping** (app/services/cerfa.py:90-205, app/routers/sourcing.py:420-450, app/services/agents/redaction.py:27-59 (needs refactor))
  - Problème : NO questionnaire auto-completion or structured field filling. The system generates static CERFA PDFs (DC1, DC2, ATTRI1) with pre-filled company data (app/services/cerfa.py:90-205), but: (1) Zero parsing of customer RFP/RFI/DDQ spreadsheets or structured question lists; (2) Zero mapping of DCE content to RFP/RFI questions (e.g., 'capacity' section → RFP question 12); (3) Zero cell-by-cell auto-completion logic; (4) No tracking of which answers come from DCE vs company profile vs defaults. TenderCrunch's differentiator: bi-modal output (memory long + cellule-par-cellule RFP answering with source attribution).
  - Reco : Build a questionnaire engine: (1) Create app/services/questionnaire.py (NEW) with SchemaParser class to ingest RFP/RFI/DDQ files (Excel, PDF forms, JSON); (2) Normalize questions to a QuestionType enum (capacity, experience, methodology, compliance, price, delivery, personnel, insurance, etc.); (3) Create AnswerMatcher class that maps DCE clauses/extracted data (via RAG) → RFP questions with confidence scores; (4) Implement FieldFiller agent that generates natural-language answers contextualized to each question + source citations; (5) Add endpoint POST /api/sourcing/{project_id}/autofill-questionnaire that takes uploaded RFP + returns {'question_id': {'answer': str, 'confidence': float, 'sources': [...]}}; (6) Expose /api/sourcing/{project_id}/questionnaire for UI to display fillable grid with AI-suggested answers + confidence indicators. Example: RFP row 'Describe your team structure' → FieldFiller extracts from DCE 'moyens humains' section + company qualifications → returns answer 'Équipe de 12 ingénieurs spécialisés...' [source: mémoire technique p.2].
- **[P2] Compliance Checker Agent (4th Agent)** (app/routers/sourcing.py:420-450, app/services/agents/redaction.py:61-143)
  - Problème : No post-redaction compliance validation. The Redacteur generates memoire + CERFA but no agent verifies: (1) All mandatory clauses answered (from DCE 'pieces_requises'); (2) Consistency between CERFA sections (D,E,F) and company profile; (3) Risk clauses acknowledged (penalites, garantie, delai) vs company thresholds; (4) Response completeness score (e.g., 87% fields filled, 3 critical gaps flagged). TenderCrunch: conformance checker catches missing DUME sections, unsigned attestations, mismatches.
  - Reco : Create app/services/agents/compliance.py (NEW) with a ComplianceChecker class: (1) Parse DCE 'pieces_requises' list; (2) Cross-check generated documents against list (DC1, DC2, ATTRI1, memoire, national form if EU); (3) Validate CERFA section consistency (e.g., company name in D matches in company profile, all co-traitants listed in E are in groupement); (4) Verify risk clauses (loop through ai_analysis['details']['clauses_risque'], confirm user acknowledged each 'eleve' level); (5) Calculate completeness score (fields_filled/fields_total); (6) Return ComplianceReport with {missing_docs: [], consistency_errors: [], risk_gaps: [], completeness_pct: int, must_fix: [], nice_to_have: []}. Add endpoint POST /api/sourcing/{project_id}/validate-compliance returning report + auto-generated remediation checklist. Call this before /documents returns download link.
- **[P2] Long-Form Memory vs Questionnaire Bi-Modal Output** (app/services/agents/redaction.py:27-59, app/routers/sourcing.py:427-445)
  - Problème : Redacteur generates single-format output: memoire Markdown (app/services/agents/redaction.py:27-58) + CERFA PDFs. No dichotomy between: (1) 'long-form strategic memory' (narrative, continuous prose answering 'how will you execute?'); (2) 'questionnaire-driven cell-by-cell responses' (structured, indexed, with source traceability). A single RFP might need both: marketing material (memoire) + technical detail sheets (RFP Q&A grid).
  - Reco : Refactor redaction agent to produce dual output: (1) Maintain existing memoire_technique as 'strategic narrative'; (2) Add new generation mode POST /api/sourcing/{project_id}/generate-questionnaire-answers: takes optional RFP template, outputs {questions: [{q_id, q_text, a_text: str, a_sources: [clause_ref], a_confidence: float, a_length_chars: int}]}. Use different temperature + length constraints for each mode (memoire: temp=0.4, ~600 words; QA: temp=0.1, ~150 words per Q). Inject RAG context differently: memoire gets full DCE context; QA gets only relevant clauses per question. File: app/services/agents/redaction.py refactor _generate_memoire_fast() into _generate_memoire_strategic() + new _generate_questionnaire_answers() + shared helper _retrieve_context().
- **[P2] Hallucination Prevention & Document Integrity Audits** (app/services/analysis.py:237-301, app/services/llm.py:29-68)
  - Problème : Anti-hallucination claims in code comments (e.g., app/services/agents/amont.py:8-11 'jamais invente', 'traçabilité') but no runtime verification or audit trail. No systematic checks: (1) Does extracted project budget match actual DCE text? (2) Are clauses_risque values cited in the document or fabricated? (3) Do generated CERFA field values come from uploaded company data or invented? No audit log of LLM inputs/outputs per call.
  - Reco : Implement strict source attestation: (1) app/core/llm_audit.py (NEW) wraps every complete/complete_json call to log {timestamp, model, input_tokens, output_tokens, input_hash, output_hash, user_id, project_id}; (2) Add SourceCitation model to track every claimed fact: {field_name, claimed_value, source_type: 'company_profile'|'dce_text'|'inferred'|'user_input', source_excerpt, confidence: float}; (3) In analyze_dce_text(), validate every extracted value against source text with regex/fuzzy matching before returning; (4) If value not found in DCE, mark as 'inferred' + lower confidence; (5) Export audit trail per project: GET /api/admin/audit/{project_id} → audit log + source attestations. Display 'confidence' badge in UI (green ✓ = found in doc, orange ⚠ = inferred, red ✗ = uncertain).
- **[P2] Dynamic Prompt Adaptation per Market Type & Country** (app/services/analysis.py:13-87, app/routers/sourcing.py:59-72)
  - Problème : System prompts hardcoded in French for French markets (SYSTEM_PROMPT in app/services/analysis.py:13-87). No per-country specialization: (1) French CERFA template (DC1/DC2) not relevant for Portugal, Poland, Italy; (2) Expected document structures differ (Spain uses ESPD, Romania has national addendums); (3) Penalty/guarantee clauses have different legal thresholds by country; (4) No prompt strategy switch for EU vs France. Code supports country param in models (app/models/__init__.py:66, User.org_id → Organization.country) but not leveraged in LLM layer.
  - Reco : Implement country-aware prompt factory: (1) Create app/services/prompts/ directory with per-country modules (fr.py, es.py, de.py, it.py, pl.py, ro.py, nl.py, pt.py); (2) Each module exports SYSTEM_PROMPT specialized for country's legal/procurement norms + CERFA equivalents; (3) Move criteria thresholds to config (e.g., FR: max_penalty=10%, ES: max_penalty=5%, IT: no penalty in some sectors); (4) In analyze_dce_text(), detect country from source_url (BOAMP → FR, OJEU → country code) or user org setting; (5) Load country-specific prompt via factory function get_system_prompt(country, analysis_type='go_nogo'|'memoire'|'compliance'); (6) Test each prompt against sample DCE in that country's language. File: app/services/analysis.py refactor to call prompt_factory.get_system_prompt(country).
